Implementation

Overview

If you want to maintain complete control of the customer data capture within your site then the Flywire checkout offers the most streamlined payment experience available. Customers remain on your site and the checkout modal pop-up allows the payer to chose their payment method and complete the payment in as little as one step.

On completion of a payment the customer is returned back to your site to a specified redirection URL or through a callback function. In addition, real-time notifications can be enabled to allow you update your back-office systems with payment status information.

There are several configuration options available when using the Flywire checkout. Please take a look at the following sample page and configuration options to decide which is most appropriate for your use case. All the available configuration options are explained here.

Sample Configurations

Minimum

The following parameters must be specified as a minimum to successfully render the checkout.

The customer will be prompted to enter all payer and custom field information.

We recommended that you always implement an onInvalidInput handler to report any error messages to your customers.

var config = {
  // Set the target environment (demo, prod)
  env: "demo",

  // Set your unique code (may differ between demo and prod)
  recipientCode: "Your Portal Code",

  // Set the value of the payment to be processed
  amount: 1234.56,

  // Recommended (not required) validation error handler
  onInvalidInput: function(errors) {
    errors.forEach(function(error) {
      alert(error.msg);
    });
  },

  // Display payer and custom field input boxes
  requestPayerInfo: true,
  requestRecipientInfo: true,

  // Set the return URL where payers are redirected to on completion
  returnUrl: "https://httpbin.org/get"
};

Partial Payer Info

This sample configuration will render a checkout with information input boxes partially populated by the values in the configuration object.

If complete information is provided for either the payer information or for custom fields then thoses pages will be skipped by setting the skipCompletedSteps to true.

Once the payment is complete the onCompleteCallback handler will be called and the checkout closed rather than the window being redirected.

var config = {
  env: "demo",
  recipientCode: "Your Portal Code",
  amount: 1234.56,

  onInvalidInput: function(errors) {
    errors.forEach(function(error) {
      alert(error.msg);
    });
  },

  // Display the payer and custom information input boxes
  // skipping either page if all the information has been passed in.
  // NB. You can see this behaviour by completing the address, city and
  // country parameters in the next block.
  requestPayerInfo: true,
  requestRecipientInfo: true,
  skipCompletedSteps: true,

  // Pass in partial payer information (missing parameters can
  // alternatively be omitted rather than included with empty or
  // null values)
  firstName: "John",
  lastName: "Doe",
  email: "john.doe@flywire.com",
  phone: "+44 1234 567 890",
  address: "",
  city: "",
  country: null,

  // Pass in partial custom field information (missing parameters can
  // alternatively be omitted rather than included with empty or null
  // values)
  recipientFields: {
    booking_reference: "REF1234",
    additional_information: null,
  },

  // The callback function that is called once the payment is complete
  onCompleteCallback: function(args) {
    alert(args);
  }
};

Complete Payer Info

This sample configuration will render a checkout without showing any input boxes by setting both the requestPayerInfo and requestRecipientInfo parameters to false.

It is important to provide an onInvalidInput handler in this instance to ensure any validation errors are reported back to the payer.

In addition a nonce parameter can be set that is included in the generation of the sig parameter included in the returnUrl (and in the args supplied to the onCompleteCallback handler). We recommend that you set this to a value that uniquely identifies an entity in your platform that this payment relates to - e.g. the same value as booking_reference in the sample below.

var config = {
  env: "demo",
  recipientCode: "Your Portal Code",
  amount: 1234.56,

  // Highly recommended for this configuration
  onInvalidInput: function(errors) {
    errors.forEach(function(error) {
      alert(error.msg);
    });
  },

  // Hide the information input boxes
  requestPayerInfo: false,
  requestRecipientInfo: false,

  // Pass in complete payer information
  firstName: "John",
  lastName: "Doe",
  email: "john.doe@flywire.com",
  phone: "+44 1234 567 890",
  address: "10, Downing St.",
  city: "London",
  country: "GB",

  // Pass in complete custom field information
  recipientFields: {
    booking_reference: "REF1234",
    additional_information: "This is a sample",
  },

  // Supply a nonce to be included in the signature generation
  nonce: "REF1234",

  returnUrl: "https://httpbin.org/get"
};

Webhooks

This sample configuration includes the parameters to setup payment status webhook notifications to be sent to your servers.

Notifications are POSTed to the URL specified in the callbackUrl parameter and will contain a payload that includes the value specified in the callbackId parameter. To trigger webhook notifications both these parameters must be supplied.

As with the nonce parameter, we recommend that you set this to a value that uniquely identifies an entity in your platform that this payment relates to - e.g. the same value as booking_reference in the sample below.

There are 2 versions of callbacks currently supported.

  • Version 1 (Default)
  • Version 2

For new implementations we recommend using callbackVersion: "2" as these notifications contain additional information and status transitions.

For backwards compatibility we default to use version 1 of the callbacks 
var config = {
  env: "demo",
  recipientCode: "Your Portal Code",
  amount: 1234.56,

  onInvalidInput: function(errors) {
    errors.forEach(function(error) {
      alert(error.msg);
    });
  },

  requestPayerInfo: true,
  requestRecipientInfo: true,

  returnUrl: "https://httpbin.org/get",

  // Enable payment status notification callbacks
  callbackId: "REF1234",
  callbackUrl: "https://api.yourdomain.com/flywire-notifications",
  callbackVersion: "2"
};

Configuration Parameters

Most of the configuration parameters have been demonstrated in the sample configuration blocks above. Usage of each of these as well as some additional parameters are described in detail here.

Environment

By default our demo environment will be used if no env parameter is supplied.

Available Values

  • demo (Default)
  • prod
// Set the target environment (demo, prod)
env: "demo",

Recipient Code

The recipientCode parameter takes the 3 letter code that directs a request through to the appropriate Flywire payment portal. This may differ between your demo and production environments.

// Set your unique code (may differ between demo and prod)
recipientCode: "Your Portal Code",

Amount

In order to successfully initiate a payment and load the Flywire checkout the amount parameter must be set.

// Set the value of the payment to be processed
amount: 1234.56,

Input Validation

If any of the information passed to the checkout does not pass validation then an array of error objects will be returned to onInvalidInput callback function.

In the case where invalid payer information is passed to the checkout and requestPayerInfo or requestRecipientInfo is set to true the payer will be able to correct any invalid details within the checkout.

// Recommended but not required - include a validation callback.
// Please provide your own validation reporting feature rather
// than alerting the user and logging to the console.
onInvalidInput: function (errors) {
    errors.forEach(function(error) {
        alert(error.msg);
    });
},

Below is an example of the errors array that is passed as an argument into the function.

[
  {
    id: "email",
    msg: "Please enter a valid email address",
    msgId: "emailConstraintNotice",
  },
  {
    id: "amount",
    msg: "Required",
    msgId: "validation.required",
  },
  {
    id: "amount",
    msg: "Invalid amount format",
    msgId: "validation.invalidAmountFormat",
  },
];

Messages in the msg field will be translated into the language set by the locale parameter when initiating the checkout. Please use the msgId field to test for and handle specific error types.

You can view metadata related to all the fields required in the configuration object by checking the window.FlywirePayment.fields object in the browser console after the flywire-payment.js script has loaded.

// Pass in payer information

Request Payer Info

If you would like Flywire to capture either all or any missing or invalid payer information as part of the checkout process you should set the requestPayerInfo parameter to true.

Alternatively you should set it to false (or omit the parameter) and supply the payer information values in the configuration object.

If you set the value to false you should implment an onInvalidInput handler to display any validation errors to the payer.

Available Values

  • true
  • false (Default)

Please note that the country should be the 2-letter ISO country code

// Display the payer information input boxes
requestPayerInfo: true,

// Pass in partial payer information (missing parameters can
// alternatively be omitted rather than included with empty
// values)
firstName: "John",
lastName: "Doe",
email: "john.doe@flywire.com",
phone: "+44 1234 567 890",
address: "",
city: "",
country: "",

Alternatively all of the following fields must be supplied with values.

// Hide the payer information input boxes
requestPayerInfo: false,

// Pass in complete payer information
firstName: "John",
lastName: "Doe",
email: "john.doe@flywire.com",
phone: "+44 1234 567 890",
address: "10, Downing St.",
city: "London",
country: "GB",

// Implement an error handler

    errors.forEach(function(error) {
        alert(error.msg);
    });
},

Request Custom Info

If you would like Flywire to capture either all or any missing or invalid custom fields as part of the checkout process you should set the requestRecipientInfo parameter to true.

Alternatively you should set it to false (or omit the parameter) and supply all the recipientFields information in the configuration object.

If you set requestRecipientInfo to false you should implment an onInvalidInput handler to display any validation errors to the payer.

Available Values

  • true
  • false (Default)
// Display the custom field input boxes
requestRecipientInfo: true,

// Pass in partial recipientFields information (missing parameters can
// alternatively be omitted rather than included with empty values)
recipientFields: {
    booking_reference: ""
}

Alternatively all of the required fields must be supplied with values.

// Hide the custom field input boxes
requestRecipientInfo: false,

// Pass in complete payer information
recipientFields: {
    booking_reference: "REF1234"
},

// Implement an error handler
onInvalidInput: function (errors) {
    errors.forEach(function(error) {
        alert(error.msg);
    });
},

Read Only

If you would like any fields to be read-only (e.g. firstName, country etc..) you can use the readonlyFields array to specify which fields you would like to be un-editable.

These fields can be either the payer info or custom info fields.

// Pass in all portal specific required custom fields
firstName: "John",
lastName: "Doe",
country: "US",

requestPayerInfo: true,
requestRecipientInfo: true,

recipientFields: {
    booking_reference: "REF1234"
},

readonlyFields: [
    "firstName",
    "lastName",
    "country",
    "booking_reference"
],

Skip Steps

In order to avoid payers clicking through unnecessary steps in the checkout you can set the skipCompletedSteps parameter to true alongside the requestPayerInfo and requestRecipientInfo parameters as necessary.

This is useful if you are unsure whether either the payer details or custom field values you supply will be complete or partial.

With this set to true if all the required information has been supplied the checkout process will jump to the next step automatically.

There is also the option to skip only a certain step. You can do this by specifying the step name. For example you can skip the custom fields step if it has been completed by specifying ['requestRecipientInfo'].

Available Values

  • true
  • false (Default)
  • [step name] ('requestPayerInfo' or 'requestRecipientInfo')
 // Display the payer and custom information input boxes
 // skipping either page if all the information has been passed in.
 requestPayerInfo: true,
 requestRecipientInfo: true,
 skipCompletedSteps: true,

You should not use skipCompletedSteps if you want to give the payer the chance to edit the fields information.

Filtering

By default Flywire will show all available payment options for a given a portal during checkout.

In certain circumstances it may be desirable to restrict the methods available to your customers based on a set of rules (e.g. only accept bank transfers for payments overs a certain value, only show card payments a certain time before a booking etc).

To enable this the Flywire checkout allows you to filter the payment options offered by payment type and by payer currency. Filters are supplied in the paymentOptionsConfig parameter.

Available Payment Type Filters

  • bank_transfer
  • credit_card (Show all card payment options e.g. debit cards)
  • online (Show all online alternative payment options e.g. PayPal)

Availble Currency Filters

  • local (Only show payment options in the payers local currency e.g. only GBP options for a payer in the UK)
  • foreign (Only show payment options not in the payers local currency e.g. everything but GBP options for a payer in the UK)
  • fx (Only show payment options that are a different currency to the collection currency)
  • nonFX (Only show payment options that are in the same currency as the collection currency)
paymentOptionsConfig: {
    // Only allow local card and online payments
    filters: {
        type: ['online', 'credit_card'],
        currency: ['fx']
    }
}

Sorting

Flywire makes every effort to display the payment option to payers in the most appropriate order based on payer behaviour.

However, we recognise that you may prefer to change this order to better suit yours and your payers needs.

To enable this the Flywire checkout allows you to sort the payment options offered by payment type, payment currency and/or amount. The sort orders are supplied in the paymentOptionsConfig parameter.

Availble Payment Type Sort Options

  • bank_transfer
  • credit_card
  • online (This refers to online alternative payment options e.g. PayPal)

Availble Currency Sort Options

  • local (The payers local currency)
  • foreign (Not the payers local currency)

Availble Amount Sort Options

  • asc (ascending amount order)
  • desc (descending amount order)
paymentOptionsConfig: {
  // Show the cheapest local payment option first but prefer bank transfers
  // over cards and online methods if they are the same price
  sort: [
    { currency: ["local", "foreign"] },
    { amount: "asc" },
    { type: ["bank_transfer", "credit_card", "online"] },
  ];
}

The order in which type, currency and amount are set in the sort array affects the final sort order.

Nonce

Flywire includes a sig parameter in query string of the return URL and in the args supplied to the onCompleteCallback handler.

To include your own parameter in the generation of the signature please supply a value to the nonce parameter.

We recommend that you set this to a value that uniquely identifies the entity in your platform that this payment relates to - e.g. a booking reference, reservation number, registration id etc.

// Supply a nonce to be used in signature generation
nonce: "REF1234",

Details of how to verify the signature can be found in the verify signature section.

Return URL

Once a payment is completed the checkout modal is closed. If you have specified a value for the return URL the page will be redirect to the URL specified in the returnUrl parameter.

// Set the return URL where payers are redirected to on completion
returnUrl: "https://httpbin.org/get";

The following parameters are included in the query string of the returnUrl

ItemParameterValue
Referencereferencee.g. FLY123456789
StatusstatusPossible values are pending, success, error
Amountamounte.g. 1234.56
Payment Methodpayment_methodPossible values are bank_transfer, online
Nonce (if supplied)payment_methode.g. REF1234
SignaturesigPlease refer to the verify signature section

The pending status always returned when the payer initates a bank transfer and never returned for online payments.

An example return URL is:

http://httpbin.org/get?reference=FLY123567890&status=success&amount=1234.56&payment_method=online&_=REF1234&sig=abcdefgh

onCompleteCallback

An alternative to using the returnUrl is to specify a function as the onCompleteCallback parameter. When this is done, control is passed back to this function once the modal has closed.

You should only include a value for either the returnUrl parameter or for the onCompleteCallback parameter, never both.

onCompleteCallback: function(args) {

Extended Validation

Flywire performs quality checks on the information provided for every payment.

In cases where the information provided doesn’t pass our validation checks it will be processed but then placed in a quality check queue. To reduce the chances of payments being held in this queue the Flywire checkout validates the information provided up-front and alert payers to any potential validation errors before processing.

Should you wish to disable this feature you can set the skipMQC parameter to true.

Available Values

  • true
  • false (Default)
// Skip the quality check validation rules
skipMQC: true,

Split Payments

If you would like Flywire to automatically distribute part of the payment amount to one or more of your partners (that must also be enrolled with Flywire) you can use the payables parameter to set up split payments.

Each payable takes the amount specified in the payable away from the amount of the payment and pays it out to the specified recipient. The following parameters make up a split payment configuration within the payables array.

ItemParameterValue
Amountamounte.g. 10.00 (mandatory) - must not exceed the base amount
Recipientrecipiente.g. FWB (mandatory) - 3 letter Partner Portal code
Descriptiondescriptione.g. Commission for booking ref. 123456
// The base portal code - any remaining amount left after the payables
// distribution will be sent to this account (in this instance 70.00)
recipientCode: "Your Portal Code",

// The base (total) amount charged to the payer
amount: 100.00,

// Split payments configuration
payables: [
    {
        // Subtracted from the base of 100.00
        amount: 10.00,
        recipient: "First Partner Portal Code",
        description: "First partner payment description"
    },
    {
        // Subtracted from the base of 100.00
        amount: 20.00,
        recipient: "Second Partner Portal Code",
        description: "Second partner payment description"
    }
]

Please bear in mind the following constraints when using split payments.

  • The payables array must contain at least one element.
  • The payables array cannot include duplicate recipient’s.
  • Each recipient must be a valid Flywire 3 letter code and cannot be be the same as the base payment recepientCode.
  • The sum of all payable amount parameters must not exceed the base payment amount.

Webhook Notifications

The ​callbackUrl, callbackId are optional parameters for systems that require real-time payment status updates to be POSTed to their servers. To use callbacks you must set values for both of these parameters

The callbackUrl should be a publicly accessible endpoint.

We recommend that you set the callbackId to a value that uniquely identifies the entity in your platform that this payment relates to - e.g. a booking reference, reservation number, registration id etc.

There are 2 callbackVersion values currently supported.

  • "1" (Default) - please read v1. Notifications
  • "2" (Recommended) - please read v2. Notifications

For backwards compatibility we default to use version 1 of the callbacks however, we recommend that you use the latest version (currently version 2) as these notifications contain additional information and status transitions.

// Enable payment status notification callbacks
callbackUrl: "https://api.your-domain.com/flywire-notifications",
callbackId: "REF1234",
callbackVersion: "2"

Please refer to the webhooks documentation for details on how to process notifications.

Signature Verification

In order to verify that the parameters included in the returnUrl and the args supplied to the onCompleteCallback handler, Flywire includes a sig parameter.

This is signature is the SHA256 hash of the parameters plus a shared secret. Please contact your account manager to arrange the transfer of your shared secret.

You should only ever verify the signature on your server and you should never send your shared secret to the client’s browser.

const crypto = require("crypto");

//
// Retrieve this value from your settings
//
const sharedSecret = "1234567890123456789012345678901234567890";

//
// Retrieve these values from the returnURL query string params
//
const _ = "REF1234";
const reference = "FLY1234567890";
const status = "success";
const amount = "1234.56";
const paymentMethod = "online";
const sig = "631da0e951478c11bcb9831764de45c21a9c79d2a3b76361dd57a7127219ebcb";

//
// Regenerate the signature locally
//
const data = _ + sharedSecret + reference + status + amount + paymentMethod;
const signature = generateSignature(data);

//
// Compare and continue
//
if (signature === sig) {
  console.log("Match");
}

//
// Utility function
//
function generateSignature(data, sharedSecret) {
  const hash = crypto.createHash("sha256");
  hash.update(data);

  return hash.digest("hex");
}

<?php
//
// Retrieve this value from your settings
//
$sharedSecret = '1234567890123456789012345678901234567890';

//
// Retrieve these values from the returnURL query string params
//
$nonce = 'REF1234';
$reference = 'FLY1234567890';
$status = 'success';
$amount = '1234.56';
$paymentMethod = 'online';
$sig = '631da0e951478c11bcb9831764de45c21a9c79d2a3b76361dd57a7127219ebcb';

Cookbook

What follows is some suggested guidance on how to successfully implement the Flywire checkout. However, you should read the documentation fully and determine what is appropriate for your use case. If you need to discuss with Flywire please contact your account manager to arrange a call. This assumes you will be using version "2" of the webhook notifications.

  1. Configure the checkout with the appropriate parameters ensuring you include values for the nonce, callbackUrl and callbackId and parameters. It often makes sense to use the same value for both the nonce and callbackId. Set the callbackVersion to "2".

  2. When a payer selects a payment option and continues to the payment page an initiated notification will be sent to your callbackUrl. This includes the payment method (bank_transfer, credit_card, online etc) and additional metadata (varies between methods). For bank transfers it may make sense for you to hold a booking or reserve product as the customer still needs to send Flywire the money to our bank account. We give customers a certain amount of time to do this as indicated by the expiry date in the notification message.

  3. When the payer completes the payment then verify the signature included in either the returnUrl query string or in the args supplied to the onCompleteCallback handler. Perform this verification on your server. Use this information to show the user the appropriate messaging - do not use it to release products or sevices to your customers.

  4. Use the status and payment method information returned in the webhook notifications sent to your callbackUrl to progress any orders, release services, complete bookings etc. This can either be done on guaranteed (Flywire has the funds) or delivered (Flywire has remitted the funds) to your registered bank account. A cancelled notification will be sent to you when the booking expires.
